Client Report Portal V1

A FastAPI, SQLite, and lightweight React prototype for managing many client households, entering quarterly financial data, calculating report totals, and generating local PDF downloads for SACS and TCC reports.

No Auth Prototype PDF Download V1

Project Context

This project is for a small financial planning team that prepares quarterly client meeting reports for high net worth households. Today, the team gathers information from several places, manually calculates totals, and assembles two polished client-facing report documents. The current workflow can take most of a day per client and is vulnerable to manual math and alignment mistakes.

V1 replaces that workflow with one internal web portal. The team will keep entering live balances manually, but the portal will organize each client household, calculate the required totals, generate the SACS and TCC reports as PDFs, and keep a local history of generated reports.

Product Direction

The portal UI should be modern, clean, and operational, using the embedded WealthPortal layout reference as the design direction. The generated PDFs should remain faithful to the embedded SACS and TCC report references, with only light polish and fixed alignment.

Domain Definitions

Term Meaning in This App
SACS Simple Automated Cashflow System. A cashflow report showing monthly inflow, outflow, excess cashflow, private reserve, and investment account information.
TCC Total Client Chart. A net worth overview showing retirement accounts, non-retirement accounts, trust assets, liabilities, and summary totals.
Inflow After-tax income deposited into the client's primary inflow account.
Outflow The agreed monthly expense amount transferred to the client's spending account.
Private Reserve High-yield savings or reserve account where excess cash accumulates.
Trust Usually represented by the household's residence or other trust-held asset. Included in grand total but excluded from non-retirement total.
Liabilities Mortgages, auto loans, or similar debts. Displayed separately and not subtracted from net worth in V1.

Error Handling

The MVP should handle errors gracefully. User-facing errors must explain what happened, what the user can do next, and whether their entered data was preserved. Technical details should be logged for debugging but not exposed in the interface.

Error Case User-Facing Behavior System Behavior
Missing Required Report Data Highlight incomplete fields and prevent PDF generation until they are completed. Return structured validation errors keyed by field so the React form can focus the right inputs.
Invalid Numbers or Dates Show inline messages such as "Enter a valid dollar amount" or "Enter a valid report date." Validate on both frontend and backend before calculations or persistence.
Client or Report Not Found Show a clear empty/error state with a safe link back to the client list. Return a 404 response with a stable error code and log the missing record id.
PDF Generation Failure Show that the report could not be generated and keep the report run in draft so the user can retry. Do not create a completed report record unless both file generation and metadata persistence succeed.
PDF File Missing from Local Storage Show "This saved PDF is unavailable" instead of a broken download link. Log the missing file path and keep report metadata available for investigation.
Database Write Failure Show that changes could not be saved and ask the user to retry without losing the current form state. Rollback the transaction and log the failed operation.
Unexpected Server Error Show a generic recovery message and avoid exposing stack traces or internal paths. Return a 500 response with a request id and write the full error to server logs.

Data Sources and Entry

V1 does not connect to external financial systems. All current balances and values are entered by the team after they collect them from their existing sources.

Source V1 Treatment Examples

Implementation Phases

Phase 1 - Foundation

  1. Scaffold FastAPI backend and Vite React frontend.
  2. Configure SQLite, local file storage, and development scripts.
  3. Create seed data for many fake clients, including realistic household/account variation.

Phase 2 - Data Model and APIs

  1. Add models for clients, household members, accounts, liabilities, trust assets, report runs, report inputs, and generated reports.
  2. Add indexes for client name, status, next meeting date, last report date, and report run date.
  3. Build server-side pagination, search, sorting, and filtering APIs.

Phase 3 - Portal UI

  1. Build the clients command center for hundreds of records.
  2. Build the client detail screen based on the embedded portal layout reference.
  3. Add report readiness indicators and report history download actions.

Phase 4 - Report Workflow

  1. Create a guided report-run form grouped by SACS and TCC fields.
  2. Prefill static values from the client profile.
  3. Show last known dynamic values and validate required fields before generation.

Phase 5 - Calculations and PDFs

  1. Implement and test all deterministic calculation rules.
  2. Generate SACS page 1, SACS page 2, and TCC PDFs from fixed layouts.
  3. Support up to six retirement accounts per spouse, six non-retirement accounts, and three liabilities.

Phase 6 - Verification

  1. Run calculation unit tests and report-generation smoke tests.
  2. Visually inspect generated PDFs against the embedded report references.
  3. Check desktop and smaller viewport behavior for the portal UI.

Acceptance Criteria

  • The client list remains usable with hundreds of seeded clients through server-side search, filters, sorting, and pagination.
  • A user can open a household and see members, linked accounts, liabilities, net worth summary, and report history.
  • A user can start a report run, complete required SACS and TCC inputs, and generate PDFs without missing required data.
  • User-facing errors are clear, recoverable, and preserve entered form data wherever possible.
  • All calculation rules are covered by unit tests and match the stated business rules.
  • Generated reports are saved locally and can be downloaded again from report history.
  • The portal UI is modern and professional, while generated PDFs remain close to the existing SACS and TCC report layouts.

Technical Shape

  • FastAPI provides JSON APIs and PDF generation endpoints.
  • SQLite stores clients, report runs, and generated report metadata.
  • React renders the portal, client list, client detail view, and report workflow.
  • Generated PDFs are written to local storage and referenced from report history.
  • Backend responses use stable error codes and validation payloads that the UI can display cleanly.
  • No authentication, external APIs, or cloud file sync are included in the prototype.

Report Outputs

  • SACS page 1: monthly inflow, outflow, excess transfer, and private reserve diagram.
  • SACS page 2: private reserve balance, investment account balance, and reserve target context.
  • TCC: household members, retirement accounts by spouse, non-retirement accounts, trust value, liabilities, and summary totals.
  • PDF layouts use fixed positioning to avoid accidental movement or misalignment.

Calculation Rules

  • SACS excess equals inflow minus outflow.
  • Private reserve target equals six months of expenses plus deductibles.
  • Retirement totals are calculated per spouse.
  • Non-retirement total excludes trust assets.
  • Grand total includes retirement, non-retirement, and trust values.
  • Liabilities are displayed separately and are not subtracted from net worth.

Scale Requirements

  • Server-side client search and pagination.
  • Filters for status, tier, next meeting date, last report date, missing data, and assigned team member.
  • Quick actions from the client list: view, start report, download latest, and edit.
  • Report readiness indicators before generation.
  • Local report files organized by client and quarter.

Seed Data

  • Use fake client households only.
  • Include enough records to test search, filters, pagination, and report statuses.
  • Include varied household types, account counts, trust values, and liabilities.
  • Include several prior report records so report history can be tested.

Important Boundary

The portal can improve the team's workflow and visual organization. The generated reports should not redesign the client-facing SACS or TCC materials beyond alignment fixes and light polish.
